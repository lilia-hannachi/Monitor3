RPKI Monitor 2.0

Overview

The adoption of Resource Public Key Infrastructure (RPKI) is growing. RPKI facilitates the validation of Border Gateway Protocol (BGP) announcements and assists in preventing Internet route hijacks and other routing anomalies. The relying parties (RPs) who make use of RPKI data expect it to be fully reliable but the global RPKI is subject to daily churn, human errors, system comprise, and potentially malicious manipulations. It is useful to monitor the dynamics of the RPKI and their impact on Internet routing. Such a monitor can give significant feedback to the RPs which include corporations, network operators, government agencies, and educational institutions. For this purpose, we have developed a Big Data Analytics software platform for monitoring and analyzing both RPKI repository data and BGP routing data collected by dozens of collectors world-wide. The platform makes use of the latest Big Data tools such as Hadoop, Pig, and MongoDB. It incorporates BGP route validation using RPKI Route Origin Validation (ROA) objects. It is essentially an advanced and high-performance monitoring and analysis tool that presents many insights related to the dynamics of BGP routes (vis-a-vis Validity status) and RPKI certificates and ROA objects.

For the global RPKI validation analysis we present set of analyses that focus on comparing RPKI Route Origin Attestations (ROAs) to BGP routing announcements.
The graphs below represent snapshots of RPKI validation results for both IPV4 and IPV6 unique Prefix/Origin pairs.
The Prefix/Origin pairs are extracted from BGP table dumps from more than twenty RouteViews Collectors. The table dumps are downloaded four times per day (every six hours). All unique Prefix/Origin pairs observed in a given time are validated against the global RPKI using the algorithm described in [RFC 6811].
As described in RFC 6811, Prefix/Origin validation results in one of three outcomes: not found, valid, and invalid. The validation result not found is also known as unknown [RFC 6483].

The graphs below represent snapshots of RPKI validation changes for both IPV4 and IPV6 unique Prefix/Origin pairs.
Prefix/Origin RPKI validation change results in one of six outcomes:
  • Not Found to Valid: The prefix/origin pairs switch from not found (the prefix in BGP item was not covered by any prefix in ROA) to valid (there exists a ROA item such that the prefix in BGP item is covered by the prefix in ROA. Max-Length satisfied and AS Match)
  • Not Found to Invalid:The prefix/origin pairs switch from not found (the prefix in BGP item was not covered by any prefix in ROA) to invalid (there exists a ROA item such that the prefix in BGP item is covered by the prefix in ROA. either Max-Length Exceeded, or AS Mismatch, or both)
  • Valid to Not Found: The prefix/origin pairs switch from valid (there was a ROA item such that the prefix in BGP item was covered by the prefix in ROA. Max-Length satisfied and AS Match) to not found (the prefix in BGP item is not covered by any prefix in ROA)
  • Valid to Invalid: The prefix/origin pairs switch from valid (there was a ROA item such that the prefix in BGP item was covered by the prefix in ROA. Max-Length satisfied and AS Match) to invalid (there exists a ROA item such that the prefix in BGP item is covered by the prefix in ROA. either Max-Length Exceeded, or AS Mismatch, or both)
  • Invalid to Not Found: The prefix/origin pairs switch from invalid (there was a ROA item such that the prefix in BGP item was covered by the prefix in ROA. either Max-Length Exceeded, or AS Mismatch, or both) to not found (the prefix in BGP item is not covered by any prefix in ROA)
  • Invalid to Valid: The prefix/origin pairs switch from invalid (there was a ROA item such that the prefix in BGP item was covered by the prefix in ROA. either Max-Length Exceeded, or AS Mismatch, or both) to valid (there exists a ROA item such that the prefix in BGP item is covered by the prefix in ROA. Max-Length satisfied and AS Match)

RPKI Data Analysis


The provided tool allows users to monitor and inspect RPKI Objects. RPKI Monitor 2.0 uses data generated by RPKI MIRO (https://github.com/rpki-miro).
For a given RPKI object (.CER, .ROA, .MFT, .CRL), the validation can yield three different results:
  • Valid: The object conforms fully to RFC specifications.
  • Warning: There exists a path of valid certificate and this object is treated as valid. However, some inconsistencies with the specification exit. They are minor and not problenatic from a security point of viw.
  • Invalid: The object considered invalid due to there is no path of valid certificates to the trust anchor, the signature on the object can not be verified using its supposed parent certificate, the object expired.
